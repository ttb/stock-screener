Full-stack platform · FastAPI · React · Celery · Postgres · Redis

A screening engine built like clean architecture.

Beneath the CANSLIM and Minervini screeners sits a disciplined ports-and-adapters backend: HTTP routes and Celery tasks are just two drivers of the same use cases, which depend only on abstract domain ports — never on SQLAlchemy, Redis, or Celery directly. A composition root wires the concrete adapters in. Everything that scales — queues, locks, caches, schedules — is partitioned per market.

02

Hexagonal layering

The backend is organized into four concentric layers. Dependencies point inward: outer layers know about inner ones, never the reverse. The domain/ core is pure Python — it imports no framework — so the business rules can be tested and reasoned about in isolation.

Presentation
api/v1 · schemas

Thin FastAPI route handlers. Validate Pydantic DTOs, check session auth & feature gates, resolve a use case from the DI container, and return its result. No business logic lives here.

Why the Unit of Work matters SqlUnitOfWork is a context manager that opens one SQLAlchemy session and hands every repository (scans, scan_results, universe, …) that same session. So a use case can write across multiple repositories inside a single transaction — commit once, or roll everything back on exception. The use case only sees the abstract UnitOfWork port; the SQL details stay in infra/.
03

The async backbone

Celery is split into a general compute queue plus three families of per-market queues. A single global data_fetch worker (concurrency 1) serializes all external API calls to respect rate limits, while market-scoped user_scans and market_jobs workers run in parallel across markets.

celery
concurrency 2

General-purpose compute — finalization, cleanup, misc tasks.

data_fetch_*
1 global worker · conc. 1

All external fetches (yfinance, fundamentals, universe). Serialized so providers are never hammered.

user_scans_*
per-market · conc. 1

User-initiated bulk scans. US runs while HK runs — independent workers, no contention.

market_jobs_*
per-market · conc. 1

Breadth, group rankings, feature snapshots — the scheduled per-market compute.

Weekly refresh times (full price & fundamentals) are configurable defaults read from settings; market-close and universe times are fixed crontabs.

Worker lifecycle safety Celery signals keep state clean across the fork boundary: worker_process_init disposes the inherited SQLAlchemy engine post-fork; worker_ready clears stale data-fetch locks (held >30 min); worker_shutting_down closes DB connections. Task time limit is 24h to accommodate a 9,600-stock full scan.
04

Frontend architecture

A React 18 + Vite SPA. Server state lives entirely in TanStack React Query (caching, polling, prefetching); global UI state uses React Context (no Redux). Pages lazy-load, and the axios client owns the /api prefix so route code never hardcodes it.

Routing & shell
App.jsx · React Router 6

Provider stack: QueryClient → ColorMode → MUI Theme → Runtime → StrategyProfile → Pipeline. Secondary pages are React.lazy + Suspense; routes feature-gate on runtime capabilities.

Data layer
api/client.js · React Query v5

Axios with baseURL=/api (Vite proxies to :8000 in dev). 5-min staleTime, 30-min gc; scan status & theme pipelines poll via refetchInterval.

Results table
ResultsTable.jsx · TanStack Table v8

Virtualized (react-virtual, 48px rows), sortable, with inline sparklines and drill-down to SetupEngineDrawer and the chart modal.

05

AI & discovery subsystems

Two LLM-powered subsystems sit alongside the screeners. A provider-abstracted LLM service (LiteLLM) with per-use-case presets and key rotation; a conversational assistant bridged to an external Hermes agent; and a theme-discovery pipeline that mines unstructured content into clustered market themes.

LLM
LLMService
services/llm/llm_service.py

One LiteLLM-backed interface over Groq, Minimax & Z.AI. Per-use-case presets pick a primary model + fallback chain; _call_with_fallbacks() rotates keys and fails fast on quota (TPD) exhaustion.

  • Chatbot/research → Groq
  • Theme extraction → Minimax → Z.AI
  • Streaming + function-calling tools
Assistant
AssistantGatewayService
assistant_gateway_service.py

Backend gateway to the Hermes agent. Streams chat, executes ~13 internal tools via MarketCopilotService (max 3 round-trips), and extracts cited references. Conversations persist to Postgres.

  • Web research via Tavily / Serper
  • Internal vs. web reference dedup
Themes
Theme discovery
theme_extraction_service.py

Ingest RSS/news/social → LLM extracts themes + tickers → embedding-based clustering dedups into canonical ThemeClusters with match-score provenance. A lifecycle state machine promotes / dormantizes themes.

  • Multilingual (translate + replay metadata)
  • Velocity & correlation metrics
Signals — computed, not fetched Technical signals come from TechnicalCalculatorService, which derives RSI(14), ATR(14), SMA distances, performance windows, volatility and relative volume purely from cached price data — no external calls. This is why the platform can screen 9,000+ symbols from cache without any live API calls.
06

Deployment & operations

Deployment is layered Docker Compose: a base file for local dev, a prod overlay adding resource limits & log rotation, and an HTTPS overlay adding Caddy with automatic Let's Encrypt. Markets are opt-in via Compose profiles, so a homelab can run US-only while a VPS runs all eleven.

Local / dev
docker-compose.yml

Postgres + Redis + backend + frontend(nginx) + celery (general, datafetch, beat). Access over plain HTTP.

Homelab / prod
+ docker-compose.prod.yml

Adds CPU/memory reservations & limits, json-file log rotation, and celery inspect ping health checks.

VPS / HTTPS
+ docker-compose.https.yml

Caddy reverse proxy on 80/443, auto Let's Encrypt + renewal, HSTS, zstd/gzip. Driven by $DOMAIN.

07

The principles that repeat

Five decisions echo through every layer. Recognizing them is the fastest way to navigate the codebase.

Depend on ports, not frameworks

The domain core imports no SQLAlchemy, Redis, or Celery. Use cases speak only to abstract ports, so logic is testable and the same code runs under HTTP or Celery — only the injected adapter changes.

Partition by market

Queues, Redis locks, cache keys, RS universes, freshness checks, Beat schedules, even Compose profiles are market-scoped. One market's slowness never blocks another.

One composition root

RuntimeServices is the single, lazily-initialized place adapters bind to ports. Routes and tasks pull fully-wired use cases from it — no hidden globals scattered through the code.

Server state is React Query's job

The frontend never duplicates server data into Redux. React Query owns caching, polling and invalidation; Context holds only genuinely-global UI state.
