Backend data architecture

From raw ticks to ranked setups.

The platform turns noisy, rate-limited market feeds across 11 global markets into deterministic, rankable screening results. It does this by strictly separating fetching (slow, scheduled, serialized) from screening (fast, on-demand, cache-only) — bridged by a three-tier Redis → PostgreSQL → API cache. Every field is sourced through a versioned provider policy, and every score is gated by a data-quality policy.

01

The end-to-end flow

Two clocks drive the system. A scheduled clock (Celery Beat) refreshes the universe, prices, and fundamentals into cache. A user clock (scan requests) reads only from that cache. They meet at the freshness gate — a manual scan is rejected with 409 market_data_stale if the cache hasn't caught up to the last trading day.

Stage 1 · Ingestion

Raw data enters through provider-specific service wrappers. Because no single provider covers every market or every field, a routing policy (provider_routing_policy.py) and a field-capability registry (field_capability_registry.py) decide — deterministically and versioned — which provider supplies which field in which market. Fail-closed: unknown providers are never tried.

Field-capability registry
field_capability_registry.py

A versioned matrix declaring per-field, per-market support state:

  • supported — canonical provider is primary
  • computed — derived locally from price history
  • partial — only via a fallback provider
  • unsupported — policy excludes it (graceful degrade)
Serialization & rate limits
rate_limiter.py · data_fetch_lock.py

RedisRateLimiter uses an atomic Lua script keyed per provider (ratelimit:finviz:us). Per-market Redis locks (@serialized_data_fetch) stop two jobs hammering the same market.

  • Cross-market parallelism preserved
  • In-process fallback if Redis is down
Universe ingest
stock_universe_service.py · *_ingestion_adapter.py

Each market has an official-source adapter (HKEX, TSE, KRX, TWSE, TMX, Xetra, SGX, Bursa…). Rows carry deterministic lineage_hash / row_hash to dedupe, plus sanity floors (CN ≥ 5,217 symbols) to detect failed ingests.

03

Stage 2 · Storage & caching

The three-tier strategy — Redis → PostgreSQL → live API — exists to make screening fast and cheap while respecting brutal API rate limits. Redis (DB 2) is the hot tier; Postgres is the durable cold tier; the live API is touched only on a miss or a scheduled refresh. Keys are market-scoped: price:US:AAPL:recent.

Price cache
price_cache_service.py

Stores 5 years of OHLCV (pickled) so the Volume Breakthrough scanner and 200-day MA have history.

  • TTL 7d recent window 1,825 days
  • Incremental: fetch only dates since last bar
  • Intraday bars flagged for post-close refresh
Fundamentals cache
fundamentals_cache_service.py

60+ fields per symbol. Miss path: Redis → Postgres → Finviz/yfinance, then warms Redis.

  • TTL 7d weekly refresh cadence
  • Enriches with field_completeness_score
  • USD-normalizes for cross-market ranking
Benchmark (SPY) cache
benchmark_cache_service.py

SPY is read by every RS calculation — a classic thundering-herd risk.

  • TTL 24h trading-day aware
  • Redis SET NX EX 30 distributed lock
  • Waiters poll 0.5s; 10s timeout fallback

The freshness gate — why a scan can be refused Manual scans run cache-only: they never fall back to a live API mid-request. Before scanning, market_data_freshness.check_symbol_freshness() verifies cached prices have reached the market's last completed trading day. If not, the API returns 409 market_data_stale with per-market coverage detail (covered_symbols, oldest_last_cached_date, expected_date) rather than silently screening on stale data. It fails closed — if the trading calendar can't be resolved, the scan is rejected.

Stage 3 · Transformation

This is where cached prices & fundamentals become screening signals. The ScanOrchestrator fetches each symbol's data once, computes a shared bundle of derived indicators, then runs every requested screener over that same bundle in parallel — eliminating N×M redundant calculation.

Why "RS as percentile" matters Relative Strength isn't stock_return − benchmark_return. The data-prep layer builds a market-partitioned universe of relative performances (rs_universe_performances[period]) and ranks each stock against its own market's cohort. A US stock and a Hong Kong stock are never compared on the same RS scale — preventing cross-market distortion.

Stage 4 · Scoring & output

Aggregation and rating live in domain/scanning/scoring.py as pure, deterministic functions (no I/O). Individual screener scores combine into one composite, map to a rating tier, then pass through a data-quality gate before being written to scan_results.

Composite aggregation
calculate_composite_score()
  • weighted_average — default; mean of screener scores (equal or custom weights)
  • maximum — optimistic; best single screener
  • minimum — conservative; worst screener

Rating thresholds then apply: a pass-rate adjustment downgrades a tier when <50% of screeners pass.

Rating tiers
calculate_overall_rating()
  • Strong Buy — composite ≥ 80
  • Buy — ≥ 70
  • Watch — ≥ 60
  • Pass — < 60, or 0 screeners passing
Quality-gated rating — bad data can't rank high apply_quality_policy() reads the field_completeness_score attached during caching. Completeness < 30 forces the rating to Pass regardless of technicals; 30–60 drops one tier. The reason is recorded in quality_downgrade_reason for traceability.

Post-scan pipeline Once results are written, finalize_scan_artifacts computes industry peer metrics, logs Setup-Engine distribution, enforces scan retention, publishes a UI snapshot, and pre-warms the top-50 chart cache — so the ResultsTable, Breadth page, and Group Rankings render instantly.
06

Design principles worth internalizing

The architecture repeats a few opinionated decisions. Recognizing them makes the whole codebase legible.

Separate the two clocks

Fetching is slow, rate-limited, and scheduled; screening is fast, deterministic, and on-demand. The cache is the seam between them, and the freshness gate is the contract. A scan never waits on a network call.

Fail closed, everywhere

Unknown provider → not tried. Unresolvable calendar → scan rejected. Empty Alpha Vantage field map → no data claimed. The system prefers a visible refusal over a silent wrong answer.

Compute once, share widely

One fetch per symbol; one precomputed context (MAs, RS universe, sparklines) read by all screeners. The orchestrator is the single place that touches data per symbol.

Per-market everything

Queues, locks, cache keys, RS universes, and freshness checks are all market-scoped. US scans run while HK refreshes; markets never contend or distort each other.
